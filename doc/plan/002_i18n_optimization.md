# 代码审查与国际化优化计划

## 1. 目标
全面审查现有代码库，发现潜在的逻辑缺陷与代码质量问题，并重点修正所有的国际化 (i18n) 问题，确保应用支持多语言环境（中文/英文）。

## 2. 国际化 (i18n) 优化

### 2.1 现状分析
- 检查 `src/i18n.ts` 的配置与实现。
- 扫描前端组件 (`.vue`, `.ts`) 中硬编码的文本字符串。
- 检查后端返回的错误信息是否支持国际化或仅返回机器码/英文key。

### 2.2 执行任务
- [ ] **全局扫描**: 使用正则或工具扫描所有中文字符串和硬编码的英文提示。
- [ ] **提取词条**: 将硬编码字符串提取到 `src/locales` (或 `i18n.ts` 定义的翻译源) 中。
- [ ] **组件替换**: 将组件中的文本替换为 `$t('key')` 调用。
- [ ] **动态文本处理**: 确保带参数的动态文本使用插值语法，而不是字符串拼接。
- [ ] **后端对接**: 确保后端返回的 Error Message 对前端友好，或者前端根据 Error Code 进行翻译。

## 3. 代码审查 (Code Review)

### 3.1 代码质量
- [ ] **类型安全**: 检查 TypeScript 类型定义，减少 `any` 的使用。
- [ ] **未使用的代码**: 清理未使用的变量、导入和组件。
- [ ] **代码风格**: 统一格式化配置 (Prettier/Eslint)。

### 3.2 逻辑健壮性
- [ ] **错误处理**: 检查 API 调用的 catch 块，确保所有异常都有 UI 反馈。
- [ ] **状态管理**: 审查组件间状态共享，避免不必要的 props 透传或 store 滥用。

## 4. 进度安排
1.  **第一阶段**: 建立规范，完善 i18n 基础配置。
2.  **第二阶段**: 逐个模块替换文本 (优先 Core 功能)。
3.  **第三阶段**: 全局代码 Lint 与类型修复。
